# 基于知识图谱的股票投资系统设计需求报告

## 1 引言

### 1.1 编写目的

本需求报告用于明确“基于知识图谱的股票投资系统设计”的建设目标、功能范围、用户需求、业务流程及非功能需求，为后续系统设计、编码实现、测试验证和论文撰写提供统一依据。通过需求分析，能够进一步细化系统边界，降低开发过程中的功能偏差，提高毕业设计的实施效率与完成质量。任务书已明确本课题需要完成项目文档、软件源代码和系统实现，因此形成规范的需求报告是后续开发与验收的重要基础。

### 1.2 项目背景

随着互联网金融的发展，股票投资已成为大众广泛关注的重要领域，但股票投资知识涉及行业分类、财务指标、技术指标、投资术语及策略等多方面内容，知识点数量多、关系复杂、资料分散，初学者难以形成系统化认知。开题报告指出，知识图谱能够以“实体—关系—实体”的结构表达知识网络，而大语言模型具有较好的自然语言理解与生成能力，因此将二者结合，可以实现结构化知识展示与自然语言问答的统一，为股票投资知识学习提供智能化平台。

### 1.3 项目目标

本系统的总体目标是构建一个基于知识图谱的股票投资学习系统，主要面向股票投资初学者，完成以下目标：

1. 构建股票投资领域知识图谱，对股票、行业、财务指标、技术指标、投资概念及投资策略等知识进行统一建模。
2. 实现知识数据的管理、查询与展示。
3. 实现股票投资知识图谱的可视化交互展示。
4. 集成大语言模型接口，实现自然语言问答与辅助解释。
5. 建立用户注册、登录及基础权限控制机制。
6. 支持管理员对知识图谱数据进行维护。

上述目标与开题报告中的“研究目的”和任务书中的“主要内容”保持一致。 

### 1.4 术语说明

- **知识图谱**：一种以节点和关系组织领域知识的数据表示方式，用于表达实体及实体之间的语义联系。
- **实体**：图谱中的知识对象，如股票、行业、财务指标、技术指标等。
- **关系**：实体之间的语义连接，如“属于”“包含”“关联”等。
- **Neo4j**：图数据库，用于存储知识图谱节点及关系。
- **MySQL**：关系型数据库，用于存储用户信息、权限信息及系统业务数据。
- **大语言模型（LLM）**：用于自然语言理解、问答生成与辅助解释的模型接口。

## 2 系统概述

### 2.1 系统定位

本系统是一个**基于知识图谱的股票投资知识学习与查询平台**，不是交易系统，也不是股票价格预测系统。系统重点是帮助初学者理解股票投资知识结构，实现“可视化学习 + 关键词检索 + 智能问答 + 后台维护”的综合功能。选题情况表也明确指出，系统侧重知识组织与学习辅助功能，不涉及真实交易操作与投资收益预测。

### 2.2 建设意义

系统建设的意义主要体现在以下几个方面：

第一，促进股票投资知识的系统化学习。通过知识图谱形式组织股票、行业、指标、概念等知识，可以帮助用户形成整体认知结构，提高学习效率。
 第二，提升金融知识获取的智能化水平。用户不需要掌握复杂查询语法，仅通过自然语言提问或关键词输入即可获得知识解释。
 第三，探索知识图谱与大语言模型协同应用的工程实现路径。
 第四，增强知识展示的直观性，通过图谱可视化和交互浏览提高学习体验。 

### 2.3 系统边界

本系统的边界包括：

- 提供股票投资相关知识的学习、组织、查询和展示；
- 提供知识图谱可视化浏览；
- 提供自然语言问答功能；
- 提供用户登录注册和基础权限管理；
- 提供管理员后台数据管理功能。

本系统不包括：

- 股票交易下单；
- 实时行情交易撮合；
- 收益预测与量化交易策略执行；
- 高风险金融决策自动生成。

## 3 用户需求分析

### 3.1 用户角色划分

根据任务书和题目要求，系统用户可划分为两类：

#### 3.1.1 普通用户

普通用户是股票投资初学者或知识学习者，主要需求包括：

- 注册和登录系统；
- 浏览股票投资知识图谱；
- 通过关键词搜索知识；
- 查看实体详情及关联关系；
- 使用自然语言提问并获取知识解释。

#### 3.1.2 管理员

管理员负责维护知识数据，主要需求包括：

- 登录后台管理系统；
- 新增知识实体；
- 修改实体属性；
- 删除无效或错误数据；
- 维护实体之间的关系。

任务书中已经明确提出系统需支持用户登录与权限管理，并支持管理员对知识图谱数据进行新增、修改与删除。

### 3.2 用户核心痛点

1. 股票投资知识点多而散，难以系统学习。
2. 行业、公司、指标和概念之间的关系难以直观看懂。
3. 初学者缺少专业术语基础，无法高效检索信息。
4. 传统网页展示方式难以呈现知识之间的复杂联系。
5. 单纯依赖搜索引擎容易获得碎片化答案，缺少结构化解释。

这些问题与开题报告中提到的“知识结构复杂、内容分散、难以构建系统化知识体系”的背景完全一致。

## 4 功能需求分析

### 4.1 功能总览

根据任务书和开题报告，系统应至少包含以下核心功能模块：

1. 用户登录与权限管理模块
2. 股票投资知识图谱展示模块
3. 股票投资知识查询模块
4. 智能问答模块
5. 知识图谱可视化交互模块
6. 数据管理模块

 

------

### 4.2 用户登录与权限管理模块

#### 4.2.1 功能描述

系统应支持普通用户注册、登录、退出登录，并根据身份区分普通用户和管理员权限。

#### 4.2.2 具体需求

- 用户可输入用户名、密码进行注册；
- 用户可使用账号密码登录系统；
- 登录成功后系统保存用户会话状态；
- 未登录用户可限制访问部分页面或功能；
- 管理员拥有后台数据管理权限；
- 普通用户仅具备浏览、检索和问答权限。

#### 4.2.3 输入输出

- 输入：用户名、密码等信息；
- 输出：登录结果、注册结果、权限校验结果。

#### 4.2.4 需求意义

通过用户身份识别与权限控制，保障系统基础安全，并为后续后台数据管理提供权限隔离。任务书中已明确要求“用户注册与登录”和“基本的用户权限管理”。

------

### 4.3 股票投资知识图谱展示模块

#### 4.3.1 功能描述

系统应以知识图谱方式展示股票投资领域中的核心实体及其关系，帮助用户直观理解知识结构。

#### 4.3.2 展示内容

图谱中可包括但不限于以下实体：

- 股票
- 行业
- 公司
- 财务指标
- 技术指标
- 投资概念
- 投资策略

图谱中关系可包括：

- 股票属于某行业
- 股票对应某公司
- 指标与股票相关
- 指标与概念相关
- 策略使用某类指标

这些建模对象来源于开题报告中提出的知识图谱建模目标。

#### 4.3.3 具体需求

- 系统能够加载知识图谱数据；
- 支持图谱节点和关系线展示；
- 支持不同类型节点使用不同图形或颜色区分；
- 支持展示节点名称；
- 支持根据用户操作动态刷新图谱内容。

------

### 4.4 股票投资知识查询模块

#### 4.4.1 功能描述

用户通过关键词搜索股票投资知识，系统从知识图谱中检索相关实体并展示结果。

#### 4.4.2 具体需求

- 支持按关键词查询股票、行业、指标等实体；
- 支持展示实体基本信息；
- 支持展示实体相关联的节点与关系；
- 支持点击搜索结果进入详情页或图谱定位；
- 支持模糊匹配或部分匹配。

#### 4.4.3 示例场景

用户输入“市盈率”，系统返回：

- 市盈率实体信息；
- 市盈率定义、说明；
- 与其相关的股票或投资概念；
- 相关图谱关系展示。

任务书明确指出系统应支持“股票投资知识查询”，并能展示详细信息及关联关系。

------

### 4.5 智能问答模块

#### 4.5.1 功能描述

系统集成大语言模型技术，实现用户通过自然语言提问获取股票投资知识解释。

#### 4.5.2 具体需求

- 用户输入自然语言问题；
- 系统识别问题中的核心知识点；
- 系统从知识图谱中检索相关结构化数据；
- 将图谱查询结果与问题一起提交给大语言模型；
- 返回通俗易懂的知识解释；
- 对无法识别的问题给出合理提示。

#### 4.5.3 设计原则

根据开题报告中提出的解决思路，智能问答宜采用“图谱查询 + 大语言模型解释”的方式，即先用知识图谱提供事实，再由语言模型负责生成自然语言回答，以提升准确性与可解释性。

#### 4.5.4 示例场景

用户提问：“什么是 K 线？它和技术分析有什么关系？”
 系统流程为：

1. 识别“K线”“技术分析”；
2. 查询图谱中相关实体及关系；
3. 将查询结果交给大语言模型生成解释；
4. 向用户返回答案。

任务书也已明确要求系统具备智能问答功能。

------

### 4.6 知识图谱可视化交互模块

#### 4.6.1 功能描述

系统应支持图谱可视化的动态交互，使用户在浏览图谱时具备更好的探索体验。

#### 4.6.2 具体需求

- 支持节点点击；
- 支持节点拖拽；
- 支持图谱缩放；
- 支持高亮当前节点及其关联关系；
- 支持通过搜索定位到指定节点；
- 支持局部关系展开。

#### 4.6.3 需求来源

任务书中明确提出图谱页面需要支持“节点点击、缩放、拖拽”等交互；文献综述也指出图谱可视化应支持搜索、高亮、多层级信息展现等功能，以增强知识理解效果。 

------

### 4.7 数据管理模块

#### 4.7.1 功能描述

管理员可在后台对知识图谱数据进行维护，保证系统数据的准确性和可扩展性。

#### 4.7.2 具体需求

- 支持新增实体；
- 支持修改实体属性；
- 支持删除实体；
- 支持新增关系；
- 支持删除关系；
- 支持查看实体及关系列表；
- 支持按条件筛选和搜索。

#### 4.7.3 需求意义

由于系统所用数据来自公开金融数据、基础教学资料和文献资料，系统需要具备对知识内容进行整理、修订和扩展的能力。任务书明确要求对知识图谱数据进行后台维护。

------

## 5 非功能需求分析

### 5.1 性能需求

- 普通查询操作响应时间应保持在可接受范围内；
- 图谱节点数量在基础毕业设计规模下应能正常加载与操作；
- 智能问答在调用外部模型接口时应支持合理等待与错误提示；
- 系统应支持常规教学演示与毕业答辩场景下的稳定运行。

### 5.2 可用性需求

- 页面布局应简洁清晰，便于初学者使用；
- 图谱展示应直观、易理解；
- 搜索与问答入口应明显；
- 错误信息提示应明确，不应仅返回技术异常代码。

### 5.3 安全性需求

- 登录密码应进行基础加密存储；
- 管理功能仅允许管理员访问；
- 用户输入应进行基本校验，防止非法参数提交；
- 系统接口应具备基础权限校验机制。

### 5.4 可维护性需求

- 前后端分离，方便独立开发与维护；
- 业务模块划分清晰，便于扩展；
- 图谱数据和用户业务数据分库存储，便于维护与优化；
- 数据管理功能应支持后续知识扩展。

### 5.5 兼容性需求

任务书已给出技术环境要求，系统应能在 Windows 或 Linux 环境中开发部署，并兼容 Chrome、Microsoft Edge、Firefox 等浏览器。技术栈包括 Nginx、MySQL 8.0、Neo4j、SpringBoot、MyBatis-Plus、Vue3、Element Plus 与 ECharts。

------

## 6 数据需求分析

### 6.1 数据来源

根据任务书，系统原始数据主要来源于：

- 公开股票基础信息；
- 股票投资基础知识资料；
- 股票行业分类数据；
- 技术指标及金融分析文献资料；
- 网络公开金融知识资源与教学资料。

### 6.2 数据类型

系统数据可分为两类：

#### 6.2.1 关系型数据

存储于 MySQL，主要包括：

- 用户信息；
- 管理员信息；
- 登录认证信息；
- 问答记录；
- 系统日志。

#### 6.2.2 图谱数据

存储于 Neo4j，主要包括：

- 股票节点；
- 行业节点；
- 公司节点；
- 财务指标节点；
- 技术指标节点；
- 概念与策略节点；
- 节点间语义关系。

### 6.3 数据处理要求

系统应对原始数据进行整理、清洗和结构化处理，形成标准化实体及关系数据，以支撑图谱构建和知识查询。这一点在任务书中已有明确说明。

## 7 业务流程分析

### 7.1 用户学习流程

1. 用户进入系统首页；
2. 注册或登录系统；
3. 浏览知识图谱；
4. 输入关键词查询知识；
5. 点击实体查看详情与关联关系；
6. 通过自然语言提问获取进一步解释。

### 7.2 管理员维护流程

1. 管理员登录后台；
2. 进入实体管理或关系管理页面；
3. 新增、修改或删除图谱数据；
4. 提交更新；
5. 系统更新图谱展示与查询结果。

### 7.3 智能问答流程

1. 用户输入问题；
2. 系统解析问题关键词；
3. 图数据库查询相关实体与关系；
4. 组织结构化上下文；
5. 调用大语言模型生成回答；
6. 返回解释结果给用户。

## 8 可行性分析

### 8.1 技术可行性

本课题所采用的技术路线较为成熟。前端采用 Vue3 和 Element Plus，后端采用 Spring Boot 与 MyBatis-Plus，图数据库采用 Neo4j，可视化采用 ECharts，并结合大语言模型 API 实现智能问答。这些技术均已在开题报告和任务书中明确，具备较强的实现可行性。 

### 8.2 开发可行性

开题报告指出，目前已具备 Web 开发、Spring Boot 和 MySQL 的基础，后续主要需加强 Neo4j、知识图谱建模、大语言模型接口调用和图谱可视化实现方法的学习。因此从本科毕设角度看，项目具备开发可行性。

### 8.3 应用可行性

系统面向股票投资初学者，需求明确，应用场景清晰，能够在金融知识学习与教学辅助方面发挥实际作用，也符合选题表中对项目意义的描述。

------

## 9 风险与难点分析

根据开题报告和选题表，系统主要难点包括：

### 9.1 知识图谱结构设计难

股票投资领域概念较多，如何对实体和关系进行合理抽象是主要难点。解决思路是先简化模型，再逐步扩展。

### 9.2 图数据库查询与管理难

图数据库的建模与查询方式不同于关系型数据库，需要学习 Cypher 查询语言及图数据维护方法。

### 9.3 图谱与大语言模型协同难

如何让模型既能回答自然语言问题，又能结合图谱数据进行可解释回答，是系统实现中的关键难点。开题报告已提出“图谱查询 + 大语言模型解释”的解决方案。
